conf.client.etcd.new (opts)

Create a new etcd client instance.

If etcd is started with --auto-tls (or there is other reason to don't verify server's identity on the client), set the opts.http_client.request.verify_peer option to false. It corresponds to the --insecure (-k) curl option.

If the etcd server is secured by enforcement of clients to use a client certificate, use opts.http_client.request.ssl_key and opts.http_client.request.ssl_cert options. They correspond to the --key and --cert curl options.

XXX: Add those options to the root level?

Please, note that this etcd client cannot use TLS Common Name to authenticate with the server, because the gRPC to JSON gateway does not support it.

If the authentication is enabled on the etcd server, provide opts.user and opts.password options.

etcd may be configured to don't require neither client side key & certificate, nor user ID & password.

Parameters:

  • opts (table)

    etcd client options.
    • endpoints ({string, ...})

      Endpoint URLs.
    • server_api_version (string, optional)

      This option allows to choose the server API version:

      • 'v3' (default, suitable for etcd v3.4+).
      • 'v3beta' (suitable for etcd v3.3 and v3.4)
      • 'v3alpha' (suitable for etcd v3.3 and before).

      It is not determined automatically by the client at the moment (sorry for this!).

      See etcd gRPC gateway documentation for details.

    • user (string, optional)

      A user ID to authenticate with the server.

      XXX: Not supported ATM.
    • password (string, optional)

      A password to authenticate with given User ID.

      XXX: Not supported ATM.
    • http_client (table, optional)

      HTTP client options.
    • http_client.new (table, optional)

      HTTP client instance options.
    • http_client.request (table, optional)

      HTTP client request options.

Returns:

    etcd client instance.

Raises:

See 'General API notes'.

instance.put (self, key[, value[, opts]])

Put the given key into the key-value storage.

Creates a key if it does not exist. Increments a revision of the key-value store. Generates one event in the event history.

Parameters:

  • self

    etcd client instance.
  • key (string)

    Key to put.
  • value (string, optional)

    Value to associate with the key.

    Use opts.ignore_value to leave the existing value unchanged.

    Technically it is possible to omit the value without the opts.ignore_value option, however the result may be misleading. The default string value (an empty string) will be associated with the key.

    TODO: Consider dropping the ignore_value option from the API and provide this behavious, when value == nil is given. In Lua, unlike protobuf v3, we able to distinguish nil and an empty string.
  • opts (table, optional)

    Put request options.
    • lease (integer, optional)

      Lease ID to associate with the key. Zero indicates no lease.

      TODO: When omitted -- use current lease or drops the lease on the key? Test it and update the doc.

      TODO: Describe how to obtain a lease ID.

      Note: Use 'ignore_lease' to reuse currently existing lease.
    • prev_kv (boolean, optional)

      Whether to return the previous key-value pair in the response.
    • ignore_value (boolean, optional)

      Update the key using the **existing** value: so 'value' remains the same, but 'mod_revision' and 'version' are updated.

      Returns the INVALID_ARGUMENT error if the key does not exist.
    • ignore_lease (boolean, optional)

      Update the key using the **existing** lease.

      Returns the INVALID_ARGUMENT error if the key does not exist.

     header = ResponseHeader,
     prev_kv = KeyValue (if prev_kv is set),

instance.range (self, key[, range_end], opts)

Fetch a range of keys and its values.

This methods allows to fetch key-value pairs by a key, a prefix of a key, a range of keys.

The following table summarizes different ways to indicate a range of keys.

key range_end effect
'foo' nil keys exactly matched 'foo'
'foo' 'zoo' keys in ['foo', 'zoo')
'foo' NEXT keys prefixed with 'foo'
'foo' ALL keys >= 'foo'
ALL 'foo' keys < 'foo'
ALL ALL all keys

Parameters:

  • self

    etcd client instance.
  • key (string)

    A key, a key prefix or a range start (depending of 'range_end' value).

    The special value conf.client.etcd.ALL means no lower boundary.
  • range_end (string, optional)

    Upper boundary of the range (exclusive).

    The special value conf.client.etcd.NEXT means to fetch all values prefixed with 'key'.

    The special value conf.client.etcd.ALL means no upper boundary.

    A function (key -> range_end) may be passed here.
  • opts (string)

    Range request options.
    • limit (integer)

      Limit number of returned keys. Zero means no limit.
    • revision (integer)

      Revision (a kind of point-in-time) to use for the request. Negative or zero value means using of the newest revision.

      Note: The response header always contains the newest revision. The option affects returned key-values.

      Returns the OUT_OF_RANGE error if the revision is compacted.
    • sort_order (string)

      Order for returned result:

      • 'NONE' (default, no sorting),
      • 'ASCEND' (lowest target value first),
      • 'DESCEND' (highest target value first).
    • sort_target (string)

      KeyValue field to use for sorting: 'KEY' (default), 'VERSION', 'CREATE', 'MOD', 'VALUE'.
    • serializable (boolean)

      Use serializable member-local reads.

      Range requests are linearizable by default; linearizable requests have higher latency and lower throughput than serializable requests but reflect the current consensus of the cluster.

      For better performance, in exchange for possible stale reads, a serializable range request is served locally without needing to reach consensus with other nodes in the cluster.
    • keys_only (boolean)

      Return only keys, not values.

      Since all omitted fields are set to default values (which are defined by protobuf v3), all fetched values will appear as empty strings in the response.
    • count_only (boolean)

      Return only the keys count in the range, but not key-values itself (so, response.kvs will be an empty table).

      response.count is set always, disregarding of the opts.count_only value in the request.
    • min_mod_revision (integer)

      Filter out keys with mod_revision lesser than given revision.
    • max_mod_revision (integer)

      Filter out keys with mod_revision greater than given revision.
    • min_create_revision (integer)

      Filter out keys with create_revision lesser than given revision.
    • max_create_resivion (integer)

      Filter out keys with create_revision greater than given revision.

Returns:

    Response of the following structure: 

     {
     header = ResponseHeader,
     kvs = array of KeyValue (empty table when opts.count_only is true),
     more = boolean (if there are more keys to return in the
         requested range),
     count = integer,
 }

Raises:

See 'General API notes'.

instance.deleterange (self, key[, range_end[, opts]])

Delete the given range from the key-value storage.

Increments the revision of the key-value store. Generates a delete event in the event history for every deleted key.

The meaning of key and range_end arguments is exactly same as for corresponding instance.range arguments.

Parameters:

  • self

    etcd client instance.
  • key (string)

    The first key of the range to delete.

    A string or conf.client.etcd.ALL.
  • range_end (string, optional)

    Upper boundary of the range to delete (exclusive).

    A string, conf.client.etcd.NEXT, conf.client.etcd.ALL or a function (key -> range_end).
  • opts (table, optional)

    Delete range request options.
    • prev_kv (boolean, optional)

      Whether to return deleted key-value pairs in the response.

Returns:

    Response of the following structure: 

     {
     header = ResponseHeader,
     deleted = integer (number of deleted keys),
     prev_kvs = array of KeyValue (empty table when opts.prev_kv
         is false or omitted),
 }

Raises:

See 'General API notes'.

General API notes

Integers.

Whenever a request or response field is marked as integer, the API accepts and may return two types: number or so called number64 (e.g. 1LL or 1ULL). Signed and unsigned integers are not marked explicitly. One may find precise types in the protocol description: here and here.

The subsections below describe responses parts, which are common for different responses.

Errors.

Functions of the module may raise different kinds of errors:

1. Arguments validation error (Lua string).

2. EtcdError.

3. HttpError.

4. Internal HTTP client error: say, out-of-memory during the request (box.error instance).

The subsections below describe content of those errors.

EtcdError

Fields:

  • code (integer)

    GRPC error code.

    Those codes are listed in the conf.client.etcd.error module.
  • message (string)

    What is going on wrong.
  • details optional)

    Extra data if etcd provides it.
  • code_name (string)

    Human readable name of the code.
HttpError

Fields:

  • response

    Raw HTTP response.
    • status (integer)

      HTTP code.
    • reason (string)

      Human readable interpretation of the code. Not very useful at the moment of writting ('Ok' or 'Unknown').
    • headers (table, optional)

      HTTP headers.
    • body (string, optional)

      HTTP response body.
ResponseHeader

Fields:

  • cluster_id (integer)

    ID of the cluster, which sent the response.
  • member_id (integer)

    ID of the member, which sent the response.
  • revision (integer)

    Storage revision, which is generated when the request was applied.

    TODO: Clarify when the revision is guaranteed to grow monotonically with modifications. I guess it is so for any linearizable requests (when opts.serializable is not set for a range request) or when we're sure there were no reconnect to another cluster node.
  • raft_term (integer)

    Raft term, when the request was applied.
KeyValue

Fields:

  • key (string)

    String representing the key. An empty string can not be used as a key.
  • create_revision (integer)

    Revision of last creation of this key.
  • mod_revision (integer)

    Revision of last modification of this key.
  • version (integer)

    Version of the key. A deletion resets it to zero, any modification increases it.
  • value (string)

    Value associated with the key.
  • lease (integer)

    Lease ID attached to the key. Zero indicates no lease.

    When the attached lease expires, the key will be deleted.
